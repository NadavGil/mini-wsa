The assignment has been submitted. During internal code review we identified 15 issues across severity levels. None of the bugs affect correctness of the core ingestion and alerting flows under normal single-instance, low-concurrency operation — but several would cause data integrity or reliability problems under production load. All 15 are scoped to the next sprint with fixes already prototyped.

Critical

C1
Cache-after-commit race — RepeatOffenderCache poisoned on DB rollback
Cache updated during the transaction; a rollback leaves stale counts with no corresponding persisted event.
Root cause
// enrichmentPipeline.recordInCache() is called // inside the same transaction that writes to the DB. // If saveAll() later throws, the transaction rolls // back but the cache has already been mutated. saved.forEach(enrichmentPipeline::recordInCache); eventRepository.saveAll(events); // ← rollback here // cache now poisoned
Planned fix
Split enrichment into two phases: enrichWithoutRecording() before commit, then register a TransactionSynchronizationManager.afterCommit() callback that calls recordInCache() only after the DB write is durable. Cache is never touched if the transaction rolls back.
EnrichmentPipeline.java EventIngestionService.java
C2
Synchronous ingestion — Tomcat thread held for full DB write duration
POST /v1/events/ingest blocks until persistence completes and returns 201 instead of 202 Accepted. Under burst load, Tomcat threads saturate and the service stops accepting requests.
Root cause
// EventController.ingest() List<String> ids = ingestionService.ingest(requests); return ResponseEntity.status(HttpStatus.CREATED) // 201 .body(Map.of("accepted", ids.size(), ...)); // → blocks Tomcat thread for entire enrichment + DB write
Planned fix
Add a ThreadPoolTaskExecutor (core 4, max 16, queue 1 000) with CallerRunsPolicy for backpressure. Annotate the internal persistence method with @Async("ingestionExecutor"). Controller returns 202 Accepted immediately. Timestamp validation stays synchronous so callers still get 422 immediately for bad timestamps.
AsyncIngestionConfig.java EventController.java EventIngestionService.java

Major

M1
Pagination returns wrong window for non-aligned offsets
Integer division maps any non-multiple-of-limit offset to the wrong page. offset=25, limit=20 → pageNumber=1 → rows 20–39 returned instead of 25–44.
Root cause
// SamplesService.java int pageNumber = offset / effectiveLimit; // integer division PageRequest.of(pageNumber, effectiveLimit, ...) // offset=25, limit=20 → 25/20 = 1 (truncated) // returns rows 20–39, not 25–44
Planned fix
Replace offset semantics with 0-indexed page-number semantics throughout: ?page=0 returns the first limit rows, ?page=1 the next block, etc. PageRequest.of(page, limit) is passed directly — no division. SamplesPageResponse field renamed from offset to page.
SamplesService.java EventController.java SamplesPageResponse.java
M2
Repeat-offender cache is in-memory only — breaks in multi-instance deployment
Each instance maintains its own ConcurrentHashMap window. An attacker sending requests across multiple instances is never flagged as a repeat offender.
Root cause
// InMemoryRepeatOffenderCache.java private final ConcurrentHashMap<String, Deque<Instant>> window = new ConcurrentHashMap<>(); // Per-JVM — invisible to other instances
Planned fix
Implement RedisRepeatOffenderCache using Redis sorted sets: ZADD ip timestamp, ZREMRANGEBYSCORE to expire old entries, ZCOUNT for the threshold check. Select implementation via @ConditionalOnProperty(wsa.cache.redis.enabled). InMemoryRepeatOffenderCache remains the default for dev/test.
RedisRepeatOffenderCache.java CacheConfig.java docker-compose.yml
M3
N+1 queries in alert evaluation — one DB round-trip per rule
AlertService.evaluateAll() fires one countByRuleCategoryAndTimestampOnOrAfter query for each alert rule. With 100 rules that is 100 sequential DB queries per evaluation call.
Root cause
// AlertService.evaluateAll() return alertRepository.findAll().stream() .map(rule -> evaluate(rule, now)) // ← 1 query per rule .toList(); private ... evaluate(AlertRule rule, ...) { long observed = eventRepository .countByRuleCategoryAndTimestampOnOrAfter( rule.getCategory(), windowStart); // ← N queries }
Planned fix
Group rules by windowMinutes. For each distinct window, fire one GROUP BY rule.category query that returns counts for all categories in that window. Then resolve each rule against the pre-fetched map — O(1) lookup, constant number of queries regardless of rule count.
AlertService.java EventRepository.java
M4
TOCTOU race in AlertService.define() — max-rules limit can be exceeded
count() and save() are two separate operations with no lock between them. Concurrent requests can both read count < maxRules and both proceed to save, breaching the limit.
Root cause
// AlertService.define() — no synchronization long existing = alertRepository.count(); // read if (existing >= maxRules) throw ...; // check AlertRule saved = alertRepository.save(); // write // Two threads can both pass the check before either saves
Planned fix
Add synchronized to define(). The method is called infrequently (alert rule creation is an admin action, not a hot path), so the synchronization overhead is negligible and prevents any concurrent overshoot of the limit.
AlertService.java
M5
Dead code in EventRepository — unused query methods
Several derived query methods are declared in EventRepository but never called by any service. They add noise, generate SQL at startup, and mislead reviewers about what queries the system actually uses.
Planned fix
Audit all repository methods against service call sites. Remove any method not directly invoked by a service or test. The Stats and Samples queries that are actively used are retained; unused derived finders are deleted.
EventRepository.java
M6
Rate-limit config property ignored — hardcoded value used instead
application.yml declares wsa.rate-limit.requests-per-minute but RateLimitConfig.java ignores it and hardcodes 200 req/min. Changing the property has no effect.
Root cause
// RateLimitConfig.java Bandwidth limit = Bandwidth.classic( 200, // ← hardcoded, ignores property Refill.greedy(200, Duration.ofMinutes(1))); // wsa.rate-limit.requests-per-minute in application.yml // is declared but never @Value-injected here
Planned fix
Inject the property via @Value("${wsa.rate-limit.requests-per-minute:200}") and use it in the Bandwidth.classic() call. This makes the limiter configurable per environment without code changes.
RateLimitConfig.java application.yml
M7
No observability — missing correlation IDs and Micrometer metrics
Log lines have no request identifier — impossible to trace a single request across multiple log entries. No counters for accepted/rejected events, so there is no signal for ingestion error rates.
Root cause
No RequestIdFilter exists. No MeterRegistry is injected in any service. spring-boot-starter-actuator and micrometer-core are not in pom.xml.
Planned fix
Add RequestIdFilter (@Order(MIN_VALUE)): reads incoming X-Request-Id header or generates a UUID, stores it in MDC so all log lines in the request carry requestId=... automatically, echoes it in the response. Inject MeterRegistry into EventIngestionService to increment wsa.events.accepted / wsa.events.rejected counters. Expose via /actuator/metrics.
RequestIdFilter.java EventIngestionService.java pom.xml

Minor

m1
Missing @NotBlank on ruleName — blank rule names accepted
RuleInfoDto.ruleName has no @NotBlank constraint. A blank string passes validation and is persisted, producing events with no meaningful rule name.
Planned fix
Add @NotBlank to RuleInfoDto.ruleName. The existing constraint violation handler already maps this to a 400 Bad Request.
RuleInfoDto.java
m2
clientIp not validated against IP format — any string accepted
EventIngestRequest.clientIp accepts arbitrary strings. Garbage IP values are persisted and pollute attacker-frequency statistics.
Planned fix
Add @Pattern(regexp = "^[\\d.:a-fA-F]+$") and @Size(max = 45) to clientIp in EventIngestRequest. The regex covers both IPv4 (1.2.3.4) and IPv6 (2001:db8::1) without a full parser dependency.
EventIngestRequest.java
m3
InMemoryRepeatOffenderCache is unbounded — memory leak on long runs
The ConcurrentHashMap grows without a size cap. Every unique clientIp ever seen is retained for the lifetime of the JVM process. Under attack traffic with many source IPs this can exhaust heap.
Planned fix
Add a max-entries guard (configurable via wsa.cache.max-ip-entries, default 10 000). When the map exceeds the limit, evict the oldest entry by insertion order using a LinkedHashMap with removeEldestEntry override. This is a dev-mode cache so simplicity matters over precision.
InMemoryRepeatOffenderCache.java
m4
No OpenAPI annotations on controllers — Swagger UI shows bare stubs
Controllers have no @Operation, @Parameter, or @ApiResponse annotations. The auto-generated Swagger UI at /swagger has no descriptions, parameter docs, or example payloads.
Planned fix
Annotate all controller methods with @Operation(summary = ...) and @Parameter(description = ...) for each query param. Add @ApiResponse entries for the non-200 status codes (400, 422, 429).
EventController.javaStatsController.javaAlertController.java
m5
Schema owned by Hibernate ddl-auto:update in production profile
application-postgres.yml uses ddl-auto: update. Hibernate auto-updates the schema on startup — no audit trail, no rollback path, and a known source of silent data loss on column renames.
Planned fix
Add Flyway (flyway-core — Spring Boot 3.2 ships Flyway 9.x where PostgreSQL support is built in). Create V1__initial_schema.sql from the current schema. Switch Hibernate to ddl-auto: validate so it verifies the schema matches entities but never modifies it. H2 dev profile keeps create-drop with Flyway disabled.
pom.xmlV1__initial_schema.sqlapplication-postgres.yml
m6
Missing integration test coverage for pagination and async ingestion paths
StatsSummaryIntegrationTest covers the happy path but there are no tests for page > 0 pagination, the 422 future-timestamp path, or the async completion guarantee.
Planned fix
Extend StatsSummaryIntegrationTest with: page boundary assertions (page=0 vs page=1 return different events); future-timestamp 422 test; and an async ingestion test using a SyncTaskExecutor in a @TestConfiguration to make async calls behave synchronously for assertion purposes.
