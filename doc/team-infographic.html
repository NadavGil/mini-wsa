💬 The Prompt That Started It All

Akamai Director → Claude (Cowork)
I am an Akamai director. Looking to implement a logging system. You will create a scrum team in the form of several Claude agents. We will have an architect, who will create an HLD. A tech lead who will write the tech design. Then we will have a testing specialist who will make sure to QA the system, in every deliverable. The QA specialist is also a chaos engineer who will try to break the design/system. A security specialist will look for vulnerabilities. A team lead will also be a scrum master, and will create scrum user stories, and assign them to a developer. We will have one senior developer and one junior, implementing the code. Each story completed, will be handed to the QA agent. And the developer will go and fetch more work from the lead. Planning (by architect and by tech lead) will be on Opus. Other agents will be running on Sonnet. You will implement the team lead as a workflow, fanning out concurrent tasks, whenever possible to parallelize. Assigning to either one of the developers, based on task complexity. Based on the tech design, checkpoints will be defined, each checkpoint will be pushed into git, so that we have multiple small commits, and not just one large at the end. I want the code to be testable. i.e. implement IoC for e.g. the DAL, so that a client can decide to swap to a different DB, with just a config change. Plus, DAL and IoC will yield testable code. When code complete, you will do the following loop engineering: system-wide CR → fix all findings → loop till nothing found in CR. The loop engineering will be capped at $300. It would be very interesting to see a sequence diagram with interactions between the agents. Create an infographic HTML to show the teamwork messages and collaboration. HTML as well as the HLD and design document will go under /doc in git. For any blockers, ping me, so I can decide how to unblock.

100% closed · v0.8-board-fixes
CRITICAL — 2
C1 Cache poisoning on rollback — cache.record() called inside transaction; rollback left ghost IPs in repeat-offender cache. Fix: TransactionSynchronizationManager.afterCommit() — cache only written after durable DB commit. ✓ RESOLVED
C2 Synchronous ingestion blocks I/O thread — Tomcat thread held for full DB write duration, limiting pipeline throughput. Fix: @Async("ingestionExecutor") fire-and-forget; returns 202 Accepted immediately; CallerRunsPolicy provides backpressure; timestamp validation stays synchronous for fast 4xx. ✓ RESOLVED
MAJOR — 7
M1 JPA INSERT vs MERGE ambiguity — client-supplied PK triggered merge() (SELECT + INSERT) on new entity. Fix: EnrichedEvent implements Persistable<String> — forces persist() always. ✓ RESOLVED
M2 In-memory cache broken at multi-instance scale — each pod held independent attacker state; attacks split across pods went undetected. Fix: RedisRepeatOffenderCache using sorted sets; selected via @ConditionalOnProperty(wsa.cache.redis.enabled). ✓ RESOLVED
M3 N+1 alert evaluation queries — one DB query per rule regardless of shared window. Fix: rules grouped by windowMinutes; one GROUP BY rule_category per group; composite key "windowMinutes:CATEGORY" avoids collision. ✓ RESOLVED
M4 Future-timestamp check in async path — validation ran inside async task; caller never received 422. Fix: validateTimestamps() extracted and called synchronously before async dispatch. ✓ RESOLVED
M5 clientIp not validated — any arbitrary string accepted; log injection risk. Fix: @Size(max=45) @Pattern(regexp="^[\\d.:a-fA-F]+$") on EventIngestRequest.clientIp. ✓ RESOLVED
M6 ruleName missing @NotBlank — blank or missing rule names persisted silently. Fix: @NotBlank added to RuleInfoDto.ruleName. ✓ RESOLVED
M7 Broken pagination integer division — pageNumber = offset / limit produced wrong page numbers; page 0 repeated. Fix: page-number model (?page=0); PageRequest.of(page, limit) directly. ✓ RESOLVED
MINOR — 6
Code Quality ✓ 3/3
  • m1: Dead countByClientIpAndTimestampAfter removed from repository
  • m4: synchronized AlertService.define() — TOCTOU race on rule count-check fixed
  • m5: Micrometer counters — wsa.events.accepted / wsa.events.rejected
Ops & Observability ✓ 3/3
